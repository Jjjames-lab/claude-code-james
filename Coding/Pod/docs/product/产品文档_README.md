## 🔄 文档更新机制

### 版本命名规则
```
v[主版本].[次版本].[修订版本]

示例：
- v2.1.0  → 新增播客列表功能
- v2.0.1  → Bug 修复
- v2.0.0  → 重大功能更新（设计系统 v3.0）
```

### 更新触发条件
**必须更新文档的情况**：
1. ✅ 新增功能需求
2. ✅ 设计变更（UI/UX）
3. ✅ 交互流程修改
4. ✅ 技术方案调整
5. ✅ Bug 修复（影响用户体验）
6. ✅ 性能优化

### 更新流程
```
1. 修改具体的功能需求文档
   ↓
2. 更新 CHANGELOG.md（记录变更）
   ↓
3. 更新本文件（README.md）
   ↓
4. 提交 Git（commit message 格式：docs: 更新 xxx 功能需求）
```

**按顺序阅读**：

## 📝 文档维护规范

### 文档格式
- 使用 Markdown 格式
- 代码块使用语法高亮
- 图表使用 Mermaid 或 ASCII 艺术
- 截图存储在 `参考图/` 目录

### 命名规范
```
文件命名格式：
- [功能名称]_需求文档.md
- [功能名称]_设计规范.md
- YYYY-MM-DD_[功能名称]_讨论记录.md
```

### Commit 规范
```bash
# 文档更新
git commit -m "docs: 更新播客列表功能需求（v2.1.0）"

# 新增文档
git commit -m "docs: 添加笔记功能需求文档"

# 修正文档
git commit -m "docs: 修正路由设计规范（v2.0.2）"
```
